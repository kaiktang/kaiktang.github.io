Java AbstractQueuedSynchronizer

Posted on 2022-05-05 Disqus:

AbstractQueuedSynchronizer是一个为其他同步工具提供基础设施的一个框架。

里面主要提供了三种能力：

  1. 原子的管理同步状态的能力
  2. 阻塞和解除线程阻塞的能力
  3. 维护一条线程队列的能力

管理同步状态的能力

同步状态是在AQS的一个32位整数state变量中存储的

该字段的含义是由子类进行按需赋予的，比如说：

  • 在ReentrantLock中，state被赋予的含义为锁重入的次数。
  • 在ReentrantReadWriteLock中，state被赋予的含义为：高位16位用来存储被获取的读锁的数量，低16位用来存储写锁的重入次数。
  • 等等

在介绍这三个变量之前，首先先要介绍Node的两种用途：在AQS中Node可以被用来组成两种队列：

  • sync queue：尝试获取锁的阻塞等待队列。
  • condition queue：和当前锁对象所绑定的条件变量进行了await后，需要释放锁，然后再阻塞，此时阻塞的线程会被放入condition queue中等待signal的唤醒。

condition queue中的节点在收到signal后会被唤醒，唤醒后的第一件事就是去再次获取到锁，反应到队列上，就是从condition queue中找到第一个没有被取消的节点并转移到sync queue中去等待获取锁。

介绍完这两种队列，就可以介绍上面变量的作用了，前两个变量prev和next是被用来链接sync queue的节点的，而nextWaiter是被用来链接condition queue节点的，另外特别是虽然nextWaiter并不参与sync queue的链接过程，但是在读写锁的实现中，该字段也会被利用起来被用来标识节点对应的线程是想要获取写锁还是读锁，对应的取值就是Node中剩下的那两个字段：

也许会有读者会有如下疑问：

为什么sync queue需要使用双向队列实现，而condition queue只需要使用单向队列来实现呢？

这是因为对于sync queue，它需要实现两个功能，一个是当某个节点入队之后，他需要设置它的前驱的waitStatus为SIGNAL，另一个功能是当某个节点释放写锁或者获得读锁的时候如果发现其waitStatus为SIGNAL那么去唤醒其直接后继。

第一个功能可以很方便的通过prev指针来实现，对于第二个功能，理论上也能通过prev从后往前遍历来实现，但是时间复杂度为O(n)，所以为了高效找到后继，这里添加了next指针。但是为了保证队列的高效性，sync queue的实现需要是无锁的，但是是没有无锁的双向队列的实现，所以在入队时能够保证的只有prev指针设置的原子性，而next指针是非原子的设置的。

因为如上原因，所以在AQS的实现中，next指针被作为一种优化存在，但是因为其并非原子性设置，所以能够看到在AQS获取后继的操作中会先检查next指针是否为null，如果为null，则选择从尾到头遍历。之所以这样实现是因为，在sync queue的实现中存在哨兵节点，next节点理论上是不应该为null的，所以应该再次遍历查找。

而对于condition queue而言只存在出队操作，所以维持一个next指针就足够了。且对condition queue的操作必须是在持有锁的情况下进行，所以并不存在并发问题，也就无需考虑并发控制的问题。

子类的实现

子类一般内部会有两个内部类继承于AQS，一个是公平锁实现，另一个是非公平锁实现。

公平锁一般获取锁的策略就是只要锁是free的，那么在当前线程没有入队（sync queue）的情况下，就可以抢占锁，进行插队。

非公平锁则需要严格要求，只有当前线程入队了，且为队首节点才有资格去竞争锁。

公平和非公平锁的适用场景

  • 非公平策略优点：通常来说非公平锁能够提供更高的吞吐量，这是因为非公平锁能够有效降低锁处于可用却无主状态下的时间，因为公平锁下，线程被唤醒需要一定的时间才能被重新调度，这段时间是造成锁吞吐量的下降的主要原因。

  • 非公平策略缺点：可能会造成队列中的节点的饥饿，让队首节点返回被唤醒然后竞争锁失败后再次沉睡。

  • 公平锁优点：不存在饥饿现象。另外公平锁在被用来包裹大段代码块时，其实效果更好，因为此时线程被唤醒再调用的时间相对而言比重下降，非公平锁带来的吞吐量提升的优势不明显，另一方面由于代码块很长，所以饥饿现象可能会很严重（饥饿线程需要等很长时间才能得到执行时机）。

  • 公平锁缺点：线程唤醒再调度需要一定时间，这段时间内锁是处于空闲无主状态的。

ReentrantLock

AQS的state变量含义

写锁重入的次数。

非公平策略

在非公平策略下，只要当前锁没有被占用，那么可以直接去抢（CAS）不用排队。如果当前锁已经被自己占用了，那么就累增重入次数。

公平策略

在公平策略下，保证了只有在sync queue队首的线程才有资格去获取锁。

ReentrantReadWriteLock

AQS的state含义

高位的16位，作为读锁被获得的次数。

低位的16位 ，作为写锁重入的次数。

读锁

可以看到读锁获取的主要逻辑如下：

  1. 如果当前锁正在被写者占用，那么直接退出。
  2. 如果当前非写锁占用状态，且读锁访问策略允许，那么让读锁进行重入。

其中公平还是非公平策略取决于readerShouldBlock()的实现。

公平读锁策略

在公平策略下，当前线程只有在sync queue队首时才允许去获取读锁，即使读锁之间其实并不互斥。

非公平读锁策略

在非公平策略下，只要不是下一个节点为写者在等待都可以去获取读锁。

这样设计是为了避免读者不断进入让写者饥饿。

写锁

写锁的设置和ReentrantLock一致。
